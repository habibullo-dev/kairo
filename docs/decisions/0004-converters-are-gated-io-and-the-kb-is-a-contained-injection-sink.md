# ADR-0004: Converters are gated, sandboxed I/O; the knowledge base is a contained injection sink

- **Status:** Accepted
- **Date:** 2026-07-06
- **Context phase:** Phase 4 (research + Markdown knowledge base)

## Context

Phase 4 lets Jarvis ingest files, web pages, and notes into an external Markdown
knowledge base ("LLM Wiki") and retrieve from it. Two properties of that layer are
qualitatively more dangerous than anything in Phases 1–3 and drive the whole design:

1. **Converters do I/O with the process's privileges.** MarkItDown opening an
   attacker-supplied PDF/Office file is a *read the permission gate never saw* unless
   we make it one — and a document parser is an attack surface (decompression bombs,
   pathological inputs, XML expansion) that a thread-based timeout cannot actually stop.
2. **The knowledge base is a *persistent* prompt-injection sink.** A human approves
   *a URL or a path*, not its *content*; that content is then retrieved into the
   model's context indefinitely. It is larger in volume than a `remember` (whole
   documents) and, unlike `schedule_task`, its content is retrieved as *data* rather
   than replayed as instructions — but "framing" alone does not contain it.

## Decision

**Conversion is gated like a read, twice, and sandboxed.**
- `ingest_source`'s file parameter is named `path`, and `ingest_source` is in the
  gate's default `read_tools`, so the sensitive-path floor (`jarvis.paths`) fires
  before any converter opens the file. A gate self-consistency test asserts every
  path-checked tool actually has a `path` field — a rename to `source` would silently
  disable the floor and pass every functional test.
- The `KnowledgeService` re-runs `resolve_path` + `is_sensitive_path` itself: the
  converter must not trust that the gate ran on the right field.
- Conversion runs in a **killable subprocess** (`convert_worker`) with a hard
  wall-clock kill — real cancellation, which `asyncio.to_thread` cannot provide.
  Zip-container sources have their *uncompressed* size, member count, and nesting
  pre-scanned from central-directory metadata (no extraction) before parsing.
- MarkItDown is constructed `enable_plugins=False` with no LLM client — plugins, OCR,
  and image description are off and are a future *explicit* config, never a silent
  default (pinned by a test on the constructor kwargs).
- The URL leg is SSRF-guarded (`jarvis.net`): http/https only, no UNC/`file://`, and
  loopback/private/link-local addresses are rejected on the initial URL *and every
  redirect hop*. `web_fetch` was retrofitted onto the same guard for parity.

**Provenance is derived from the database, never from content.**
- A leading YAML front-matter block in *converted* output is stripped; wiki
  front-matter is generated by the service from validated `source_ids`, and any
  front-matter in the model-supplied page *content* is dropped. Citations in query
  results are rendered from `kb_sources` columns and each excerpt is wrapped in
  explicit `--- begin/end excerpt (untrusted content) ---` delimiters, so a forged
  `[source #…]` marker inside a document is visibly *inside* a quote.

**The model cannot bypass wiki provenance.**
- `write_file` is denied under `data/knowledge/*` via a new `write_denylist` (which
  wins over the `.` allowlist), so all wiki writes go through the provenance-tracking
  `write_wiki_page`, which is jailed (no `..`/absolute/UNC/ADS/reserved-name/symlink
  escapes; `.md` only).

**Unattended ingestion is demoted and quarantined (extends ADR-0003).**
- `ingest_source`/`write_wiki_page` join `DEMOTE_ALLOW`: an interactive "always allow"
  does not extend to a background run. If a user opts a research job in via
  `scheduler.unattended_allow_tools`, its ingests land `review_status='unreviewed'` —
  excluded from query results and never superseding reviewed content — until a human
  runs `kb review`. Read-only `query`/`lint` pass through, so scheduled research works.

**`kb_chunks` and `kb_wiki_links` may be DELETEd.** They are rebuildable caches over
the markdown artifacts and wiki files — the one deliberate exception to the
never-DELETE rule that protects primary records (`kb_sources` is never deleted;
re-ingest supersedes). Documented in the migration and enforced by delete-and-replace
inside a single `transaction()`.

## Consequences

- **Upside:** the two dangerous properties are contained structurally, not by prompt
  wording. The safety suite (subprocess kill, zip-bomb refusal, wiki jail, gate
  field-consistency, `.env` deny, KB-dir write deny, quarantine, reflection firewall
  over KB results) was written and committed *before* the tools were wired — the same
  ordering discipline as ADR-0003's UnattendedGate.
- **Cost:** an autonomous nightly-research job can gather and stage but cannot promote
  to searchable knowledge without a human `kb review`. That friction is intentional —
  unattended web content becoming permanent knowledge with no human ever seeing it is
  exactly the failure this prevents.

## Alternatives considered

- **Firecrawl for web ingestion — rejected.** Its practical mode is a cloud API
  (third-party egress of every researched URL, a new secret, a new trust boundary);
  self-hosting is 5+ Docker services with the anti-bot core closed-source. The repo
  already has an ask-gated, mocked-in-tests `httpx + trafilatura` path, and MarkItDown
  covers non-article HTML. If JS-rendered sites become a real need, Firecrawl slots in
  behind the same `converters.py` boundary with no other change.
- **Docling as the default PDF converter — rejected as default, kept as an option.**
  MarkItDown is thin, fast, and in-memory; Docling brings layout/table models and
  ~2 min/100pp on CPU. It is a per-machine opt-in extra (`uv sync --extra docling`,
  `knowledge.pdf_converter: docling`) for hard-PDF corpora, not a weight every install
  carries.
- **Auto-injecting KB context into every turn (like memory auto-recall) — rejected
  for Phase 4.** Document knowledge isn't relevant to most turns the way identity
  facts are; auto-injection would double the standing injection surface with content
  the model can fetch on demand. A `knowledge.auto_context` flag is the recorded
  follow-up if Phase 5 retrieval evals show under-querying.
- **`make_front_matter` accepting content-supplied metadata — rejected.** Provenance
  must be DB-derived; content-supplied front-matter is the citation-forgery vector.

## Recorded scope decisions (from the Phase 4 addendum)

- **Graph layer is Phase 4.5 *evaluation*, not Phase 4 core.** Graphify (project/
  codebase graph), Graphiti (temporal/evolving fact graph), and Microsoft GraphRAG
  (large-corpus relationship retrieval) are named evaluation targets; Phase 4 keeps
  Markdown + SQLite + embeddings as the source of truth, with `kb_wiki_links` as the
  seam a future graph builds on.
- **Hermes-style self-improvement is deferred until after the Phase 5 eval gates.**
  Phase 4 stores lessons, research briefs, and wiki pages, but no code path feeds KB
  content into `build_system`, tool definitions, or skills — Jarvis does not rewrite
  its own prompts/tools from ingested content.
- **Odysseus is an approved product/workstation reference** for UX/operational ideas
  only (backup/restore, degraded-state reporting, provider health checks, local-first
  setup, dashboard shape). It must not alter the core loop or the permission model.
